This blog series will walk step-by-step through the implementation of a basic interpreter for a simple object-oriented language in which (nearly) all values behave like "objects" in a typical weakly & dynamically typed programming language. It aims to (among other things) introduce the reader to (or provide further insight into) parser combinators in Haskell, the semantics of "JavaScript-like" programming languages, basic compiler optimizations, performance considerations in functional programming languages and language runtimes, and other topics relating to these. Some disclaimers before we start:

- I am not a professional and do not have very deep knowledge of the subjects herein; I will do my best to communicate a level of epistemic confidence consistent with my actual experience, but this is just as much a pedagogical endeavor for me as for the hypothetical reader
- The methods and code samples shown in these blogs are not intended to be in any way "production-ready", but rather merely illustrative -- most of them will have one or many serious shortcomings, which I'll try to explicitly address when relevant
- I will avoid lingering on definitions, as there are already many excellent definitions out there for most of the terms I'll use (I'll link to those when I can); if you are still unclear on what a term means, Google it, ask your preferred LLM, or [email me](/contact)
- It is difficult to do something you have never done before while writing in detail about how you did it, without backtracking or meandering; in reality, I have peeked ahead at TODO

With this out of the way, we can discuss what we plan to do and why someone might want to do something similar. We're aiming to construct a reasonably performant interpreter for a (relatively) simple programming language with syntax and semantics most closely resembling JavaScript, but not far removed from a language like [Lua](https://en.wikipedia.org/wiki/Lua) or Python. These languages are notable mainly for being highly "dynamic" and having relatively weak type systems with little to no upfront (i.e., "compile-time") enforcement of type system invariants; and are relevant to our project also because their canonical implementations are interpreters rather than compilers. All three are ubiquitous in different niches: JavaScript in websites and web applications (including e.g., Electron applications); Python in a wide range of "backend" systems and machine learning settings; and Lua as an embedded scripting language in games and other software.

Though I am generally a proponent of strong, static type systems for enforcing invariants at compile time and ["making invalid states unrepresentable"](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/), these kinds of languages are more flexible in many ways, and useful for scripting or prototyping. They are also useful pedagogically, and more suitable for this blog post series since they allow us to circumvent e.g., having to lay out a formalization of an elaborate type system (it presents many interesting opportunities for optimization in spite of this). As for constructing an interpreter rather than a compiler: the former is often considered much simpler, especially for this kind of language -- though as we will see later, creating a reliable, correct, and *performant* interpreter is nontrivial, and there is often significant overlap between the requirements and structure of interpreters and compilers, especially as the level of performance required/the level of sophistication in each increases.

## A loose specification

Here is a rough sketch of the language I have in mind:

- variables are locally scoped by default, and can be shadowed by identical declarations in sub-blocks of the current block; variables must be declared before they are used, and this should be enforced at compile time as much as possible (when this is ambiguous, we will assume that the proper declarations have been made and raise a runtime error otherwise)
- statements, including variable declarations, should be permitted in the global scope (the scope outside of all functions)
- by default, accessing an object field/attribute that does not exist should return an undefined or null value rather than raising a runtime error; in real programming languages, this is [very much ill-advised](https://www.infoq.com/presentations/Null-References-The-Billion-Dollar-Mistake-Tony-Hoare/), but it plays nicely with the fast-and-loose design of our language and permits a convenient null-coalescing operator
- all values, including functions, lists/arrays, and "primitives" (integers, floats, booleans, etc.) should have an "object-like" interface exposed as if they were simple dictionaries, even if the underlying representation in the interpreter doesn't treat them the same way as user-constructed compound objects
- blocks in functions, conditionals, and match statements will have "Rust-like" semantics, in which most or all statements are also expressions (evaluate to a concrete value), and if the last statement (or expression) in a block returns a value, the entire block evaluates to this value
- a syntax for anonymous ("lambda") functions is available
- functions (including anonymous ones) defined inside other functions can "capture" values in their environment when they are created, creating [closures](https://en.wikipedia.org/wiki/Closure_(computer_programming)); functions should be first-class and able to be passed around as values, received by/returned from functions, assigned to variables/object fields, converted to strings, etc.
- there will be no class hierarchy or [trait](https://en.wikipedia.org/wiki/Trait_(computer_programming))/[interface](https://en.wikipedia.org/wiki/Interface_(object-oriented_programming))/[typeclass](https://en.wikipedia.org/wiki/Type_class) system built into the language, though the facilities necessary to implement it as a library should be available
- a basic tuple syntax should be available, which will desugar to a tuple object with numbered fields
- the user of the language should be able to implement "magic methods" on objects that dictate how operator overloading should behave, or allow ergonomic interaction of user-defined data structures with features like loops and conditionals; for example, `__add`, `__mul`, `__iter`, etc.

### Execution model/implementation

- the evaluation semantics should be left up to the language implementation, within reason -- in particular, the interpreter (or compiler) should decide when to perform lazy or strict evaluation, in general taking whatever choice(s) will maximize performance
- no manual memory management should be required of the programmer; where necessary, allocations and deallocations should be handled by the language runtime (probably via [garbage collection](https://en.wikipedia.org/wiki/Garbage_collection_(computer_science)))
- the implementation(s) should optimize aggressively whenever possible, including via inlining, dead code elimination, loop rewriting, function specialization, escape analysis (to eliminate heap allocations), etc.

For the parser and interpreter, I will (at least initially) be using Haskell. In addition to being a performant language (possibly the most performant of any language with its featureset and semantics) with an excellent type system, Haskell has parser combinator libraries with features that few (if any) other languages are able to replicate. Its support for writing compilers and compiler-like software is [first-class](TODO). As we will see, it is not perfectly suited to hosting a maximally efficient interpreter, but it is a perfectly good place to get started and nail down the semantics of our language (and the issues we run into will be quite instructive).

[Megaparsec](https://hackage.haskell.org/package/megaparsec) is a well-regarded monadic parser combinator library that is suited to our goals here, and which I have some (meager) experience with. Some alternatives are listed in the first few paragraphs of the [Megaparsec tutorial](https://markkarpov.com/tutorial/megaparsec.html). That said, most [parser combinators](https://en.wikipedia.org/wiki/Parser_combinator) work in much the same way: a parser is represented as a particular kind of function that consumes an input stream (e.g., characters of text) and generates structured output. Parser *combinators* are higher-order functions that build up more complex parsers from smaller building blocks (typically by either running parsers *in sequence* (failing if *any* of the constituent parsers fail to match, and otherwise assembling the result from any or all of the results of the sub-parsers) or *simultaneously* (failing if *all* of the constituent parsers fail to match, and otherwise producing an output from the first one that does match) and somehow merging their outputs. These can be designed in such a way that we can simultaneously deconstruct the raw input stream (a flat stream of tokens) while constructing arbitrary structured output (typically a parse tree or AST representing the semantic structure of the parsed text). In a language like Haskell, a monadic context can be used to implicitly track the parser state as we advance through the input; `do`-notation allows us to concisely sequence parsers while consuming their results.

(n.b.: among the reasons that other languages' canonical parsing tools are somewhat less expressive are the fact that truly first-class functions with enough compiler/runtime support to be useable in production code without incurring a brutal performance hit are a somewhat rare feature; and that other languages lack the higher-kinded types necessary to ergonomically express monads, and the supporting syntax necessary to use them effectively. These languages generally resort to [parser generators](https://en.wikipedia.org/wiki/Compiler-compiler) or runtime libraries (I like [Lark](https://github.com/lark-parser/lark)), which are sometimes more performant but lack expressivity and require substantial post-processing of the generated parse tree(s). It is somewhat of a matter of personal preference.)
